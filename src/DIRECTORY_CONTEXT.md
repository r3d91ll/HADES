# Directory-Level Semantic Context for HADES

## Philosophy
This directory structure embodies a multi-layered semantic network where:
- No node exists in isolation
- Every connection influences every other connection
- Directory-level context shapes the interpretation of all contained elements
- The whole is exponentially greater than the sum of its parts

## Semantic Layering

### Level 1: Workspace Context
The HADES workspace represents a living system for graph-enhanced retrieval augmented generation. It exists at the intersection of:
- **Theory**: Actor-Network Theory, Science & Technology Studies
- **Practice**: Code implementation, algorithm optimization
- **Research**: Continuous integration of latest papers
- **Evolution**: Self-improving through research-driven development

### Level 2: Directory Semantics

#### `/src/` - The Living Core
This directory contains the beating heart of HADES. Each subdirectory represents a different aspect of intelligence:
- **Structural Intelligence**: How components connect and communicate
- **Semantic Intelligence**: How meaning is created and propagated
- **Evolutionary Intelligence**: How the system improves itself

**Contextual Weight**: When analyzing code in `/src/`, consider:
- Interconnectedness with other modules
- Research papers that influenced the design
- Future evolution possibilities

#### `/src/pathrag/` - Path-Based Reasoning Engine
The PathRAG implementation represents our core innovation: multi-hop reasoning through knowledge graphs.

**Semantic Context**:
- Paths are not just connections but semantic journeys
- Each hop adds context that transforms meaning
- The path taken matters as much as the destination

**Directory Intelligence**: Code here should be understood as:
- Creating semantic paths, not just retrieving nodes
- Building context through traversal
- Implementing the research from PathRAG paper

#### `/src/isne/` - Inductive Shallow Node Embeddings
ISNE provides graph-aware embeddings that understand structural context.

**Semantic Context**:
- Embeddings encode not just content but relationships
- Shallow architecture allows rapid adaptation
- Inductive learning enables generalization to new nodes

**Directory Intelligence**: Components here:
- Transform isolated embeddings into graph-aware representations
- Learn from structure as much as content
- Bridge the gap between local and global understanding

#### `/src/pipelines/bootstrap/supra_weight/` - Multi-Dimensional Edge Weighting
The supra-weight system implements multi-layered relationship scoring.

**Semantic Context**:
- Edges have multiple simultaneous meanings
- Weight is not scalar but multi-dimensional
- Relationships evolve based on usage patterns

**Directory Intelligence**: Code here understands:
- Theory-practice bridges (ANT/STS connections)
- Cross-modal relationships (code-doc-config)
- Dynamic weight adjustment based on context

#### `/src/components/` - Modular Intelligence Units
Each component is a specialized intelligence module that can be composed.

**Semantic Context**:
- Components are actors in an actor-network
- They negotiate meaning through protocols
- Configuration shapes their interpretation of data

**Directory Intelligence**: When analyzing components:
- Consider their role in the larger network
- Understand their protocol-based communication
- See configuration as semantic instruction

#### `/src/research_integration/` - Self-Improvement Engine
This directory implements the research-driven development cycle.

**Semantic Context**:
- Research papers are semantic instructions for evolution
- Code improvements are hypothesis testing
- The system learns from both successes and failures

**Directory Intelligence**: This represents:
- The system's ability to understand its own limitations
- Translation of research into implementation
- Continuous semantic evolution

#### `/src/types/` - Semantic Contracts
Type definitions create the semantic boundaries and bridges.

**Semantic Context**:
- Types are not just data structures but semantic agreements
- They define how meaning flows between components
- Protocol definitions shape possible interactions

**Directory Intelligence**: Types here:
- Create the language components use to communicate
- Define the boundaries of semantic domains
- Enable safe transformation of meaning

#### `/src/config/` - Semantic Configuration
Configuration files are semantic instructions that shape behavior.

**Semantic Context**:
- Config is code in a different form
- Parameters shape interpretation of data
- Hierarchical config creates layered meaning

**Directory Intelligence**: Configurations:
- Act as runtime semantic modifications
- Enable A/B testing of different interpretations
- Create experimental semantic spaces

## Multi-Level Connectivity Principles

### 1. Vertical Connectivity (Hierarchy)
```
Workspace → Module → Component → Function → Line
```
Each level provides context for the levels below.

### 2. Horizontal Connectivity (Siblings)
```
PathRAG ↔ ISNE ↔ Supra-Weight ↔ Components
```
Peer modules influence each other's interpretation.

### 3. Diagonal Connectivity (Cross-Cutting)
```
Research → Code → Config → Types → Tests
```
Different aspects create multi-dimensional meaning.

### 4. Temporal Connectivity (Evolution)
```
Past Implementation → Current State → Future Possibility
```
History and future influence present interpretation.

## Semantic Weighting Guidelines

When the model analyzes code in any directory, it should consider:

1. **Directory Purpose**: What role does this directory play in the whole?
2. **Sibling Context**: How do peer directories influence this one?
3. **Parent Context**: What constraints/guidance does the parent provide?
4. **Child Diversity**: What does the variety of children say about purpose?
5. **Research Alignment**: Which papers influenced this directory?
6. **Evolution Trajectory**: Where is this directory heading?

## Example: Multi-Level Interpretation

Consider a function in `/src/pathrag/PathRAG.py`:

1. **Line Level**: What does this specific line do?
2. **Function Level**: What role does this line play in the function?
3. **Class Level**: How does this function serve the class purpose?
4. **Module Level**: How does this class implement PathRAG concepts?
5. **Directory Level**: How does this fit the path-based reasoning paradigm?
6. **Workspace Level**: How does this contribute to HADES goals?
7. **Research Level**: Which paper concepts does this implement?
8. **Evolution Level**: How might this improve based on new research?

## Dynamic Context Evolution

This documentation itself is part of the living system:
- It evolves as the codebase evolves
- New directories bring new semantic layers
- Research adds new interpretative frameworks
- The system can modify its own context documentation

## The Non-Existence Principle

As you noted: "all else is non-existence literally"

This means:
- Isolated nodes have no meaning
- Disconnected code is semantic void
- Understanding requires relationship
- Meaning emerges from connection patterns

Every piece of code exists only in relation to:
- Its directory context
- Its sibling components  
- Its research foundations
- Its future evolution
- Its usage patterns
- Its configuration space

## Practical Application

When our models weight connections, they should:
1. Read directory-level documentation first
2. Understand the semantic role of the directory
3. Weight connections based on multi-level context
4. Consider research-practice bridges
5. Account for evolutionary trajectory
6. Never interpret in isolation

This creates a rich semantic space where every connection is informed by every other connection, and the system truly understands itself.
